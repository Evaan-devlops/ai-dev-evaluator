## 1. Overview

The AI Dev Evaluator is a full-stack workbench for developers building LLM-powered features. It lets a developer configure real or mock data sources, send a user prompt through their application's AI pipeline, and get a structured evaluation of how good the model response is ā€” compared to what it *should* have said given the PRD of the application under test.

---

## 2. Problem Statement

A developer building an AI-powered feature (e.g. a customer support bot) has no fast way to:
- Understand which context layers improve response quality
- Compare what their app actually produces vs. what a correctly-configured evaluator would produce
- Get grounded, PRD-aware scoring with per-dimension improvement suggestions
- Test this with their own live APIs rather than mock data

## 4. Core Concepts

### 4.1 Data Mode

Every context layer can be populated from one of two sources:

| Mode | Source |
|------|--------|
| **Mock** | Pre-seeded static content (existing Phase 1 behavior) |
| **Actual Data** | Live API calls configured by the user in the Actual Data section |

A global toggle switches all layers between modes. Individual layer overrides are not required in v2.

### 4.2 The Two Sides

**Left side ā€” Context Layers:**  
The developer configures 6 context layers. In Actual Data mode, each layer's content is fetched from the API endpoint the developer configured for that layer. The assembled prompt + configured LLM produces **Model Response**.

**Right side ā€” Evaluation & Suggestion:**  
An independent agent (built from scratch, no frameworks) reads the PRD and the user input, assembles its own optimal context, generates its own response, then compares its response to the Model Response and scores it per the evaluation parameters derived from the PRD.

### 4.3 Configure LLM

The developer configures which LLM their app uses (provider, model, API key). This LLM is called to generate the **Model Response** from the assembled context layers.

### 4.4 PRD

The developer uploads or pastes the PRD of the application they are testing. Clicking **Process PRD** runs myRAG on the document and the evaluation LLM derives **Evaluation Parameters** from it. The developer can edit, add, or remove parameters.

## 5. Feature Requirements

### 5.1 Actual Data ā€” API Configuration

The developer configures one API endpoint per context layer, plus endpoints for the evaluation side:

| Slot | Purpose |
|------|---------|
| `system_prompt` | API that returns the system prompt string, OR a free-text input saved locally |
| `conversation_history` | API returning last N conversation turns |
| `knowledge` | API returning RAG-retrieved knowledge for the current query |
| `tools` | API returning available tool schemas |
| `state` | API returning live customer/session state |
| `user_input` | Free-text field (no API ā€” entered fresh each test run) |

Each API config:
- URL, HTTP method (GET/POST), optional headers, optional body template
- Optional response JSON path (e.g. `data.messages`)
- Test button to verify connectivity

### 5.2 Configure LLM

| Field | Options |
|-------|---------|
| Provider | OpenAI, Anthropic, Gemini, Custom |
| Model | Free text (e.g. `gpt-4o`, `claude-sonnet-4-6`) |
| API Key | Stored encrypted locally |
| Temperature | 0.0ā€“1.0 |
| Max tokens | Integer |

### 5.3 PRD Section

- Upload PDF/text or paste PRD text into a textarea
- **Process PRD** button ā†’ calls myRAG `/api/v1/documents/upload` + `/api/v1/queries/ask`
- Displays extracted **Evaluation Parameters** in an editable list
- User can add/remove/rename parameters
- Parameters are persisted and reused across test runs until replaced

### 5.4 Evaluation Parameters

Automatically derived from PRD by the evaluation LLM. Examples:
- `policy_accuracy` ā€” does the response follow the stated policies?
- `persona_adherence` ā€” does it match the persona defined in the PRD?
- `feature_coverage` ā€” does it address the user's actual need?

Each parameter has: name, description, weight (0.0ā€“1.0). Weights sum to 1.

### 5.5 User Input

Free-text field. Entered fresh for each test run. Represents the user's message to the application under test.

### 5.6 Model Response

Generated by:
1. Assembling enabled context layers (from Mock or Actual Data)
2. Calling the configured LLM with the assembled prompt + user input

Displayed as-is. Not scored until the Evaluation side runs.

### 5.7 Evaluation & Suggestion Side

Three-step process:

**Step 1 ā€” Build evaluation context**  
The evaluation agent:
- Reads the PRD (via myRAG retrieval)
- Gets conversation history (from Actual Data API or last 5 from local store, summarized)
- Gets tools list (from Actual Data API)
- Gets state (from Actual Data API)
- Generates its own optimal system prompt from PRD
- Assembles its own context

**Step 2 ā€” Generate reference response**  
The evaluation LLM (same configured LLM or a separate eval LLM) generates a reference response to the user input using the evaluation context. This is the "what it should have said" response. Not shown in any text box.

**Step 3 ā€” Score Model Response**  
Each evaluation parameter is scored 0ā€“5 by comparing Model Response to the reference response + PRD evidence. Quality score = weighted sum.

Score breakdown display:
- Each parameter shown as a labeled bar
- **On hover**: show insight card with:
  - Why this score was given
  - Specific evidence from PRD
  - Concrete improvement suggestion

### 5.8 Run History

Same as Phase 1 but now includes: active mode (mock/actual), LLM used, evaluation parameters version.

## 6. myRAG ā€” Document QA Service

A standalone backend-only Python service. Lives at `../myrag/` relative to this project.

### Purpose

Used by the Evaluation side to:
- Ingest the PRD document
- Answer queries like "what is the expected persona?" / "what policies apply?"
- Ground evaluation parameters in document evidence

### Key Constraints (non-negotiable)
1. No LangChain, LangGraph, or any agent framework
2. Custom bounded ReAct-style loop from scratch
3. Deterministic pipeline first; agent loop only when evidence is insufficient
4. Every answer grounded in retrieved evidence with citations
5. No frontend ā€” APIs only

### Document Modes

| Mode | Graph Type | Detection |
|------|-----------|-----------|
| Structured | TreeGraph | Heading density, section hierarchy |
| Semi-structured | TreeGraph + ChunkGraph | Mixed signals |
| Unstructured | ChunkGraph | Low heading density |
| Scanned/layout-heavy | PageGraph | Low text variance, OCR confidence |

### Retrieval Pipeline

```
Query
  ā†’ classify query type
  ā†’ lexical retrieval (BM25 / pg full-text)
  ā†’ semantic retrieval (pgvector)
  ā†’ structure-aware retrieval (TreeGraph section matching)
  ā†’ merge + deduplicate + rerank
  ā†’ sufficiency check
  ā†’ if sufficient ā†’ generate answer
  ā†’ else ā†’ bounded ReAct loop (max steps from env)
  ā†’ generate grounded answer with citations
```

### Bounded ReAct Loop

Custom from-scratch. Allowed actions: `search_lexical`, `search_semantic`, `search_structure`, `fetch_node`, `fetch_neighbors`, `fetch_parent`, `fetch_children`, `follow_reference`, `fetch_page`, `fetch_table`, `expand_same_section`. Bounded by: max steps, max context tokens, must have evidence to answer.

## 7. Data Storage

| Data | Storage |
|------|---------|
| Actual Data API configs | PostgreSQL (ai-dev-evaluator DB) |
| LLM config (key encrypted) | PostgreSQL or local encrypted file |
| PRD file + metadata | myRAG PostgreSQL |

| Run history | PostgreSQL (replacing in-memory Phase 1 store) |

## 8. Evaluation LLM

The evaluation agent uses the same LLM provider configured in "Configure LLM" by default. A separate eval model selector is a Phase 3 feature.
